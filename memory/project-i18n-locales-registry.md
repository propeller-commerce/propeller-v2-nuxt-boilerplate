**Fact:** the package-namespaced translation system has four moving
parts:

1. `app/locales/<lang>/<Namespace>.json` — one file per package
   component (ProductCard.json, CartSummary.json, AccountIconAndMenu.json,
   etc.). Edit these directly. 55 namespaces in the canonical 'en' set;
   NL has 55+ partial coverage with gaps documented in
   `app/locales/nl/_review.md`.
2. `scripts/build-locales-registry.mjs` — scans
   `app/locales/<lang>/*.json` at boot time and emits
   `app/locales/_registry.ts` with one `import` per file plus a
   `registry = { en: {...}, nl: {...} }` map.
3. `app/locales/_registry.ts` — **auto-generated, checked into git**.
   DO NOT hand-edit. Header banner says
   `// AUTO-GENERATED by scripts/build-locales-registry.mjs`.
4. `app/lib/i18n/providers/file.ts` — reads `registry` and exposes
   `getNamespace(locale, namespace) => Record<string, string>`.
   Returns `{}` for unknown namespaces (package falls back to its own
   English defaults).
5. `app/composables/useTranslations.ts` — `useTranslations('Foo')`
   returns a `ComputedRef<Record<string, string>>` keyed on
   `languageStore.language.toLowerCase()`. The package's `:labels` /
   `:productCardLabels` / `:clusterCardLabels` / etc. props accept
   this ref shape directly.

**npm scripts:** `package.json` declares:
```
"locales:build": "node scripts/build-locales-registry.mjs",
"predev": "node scripts/build-locales-registry.mjs",
"prebuild": "node scripts/build-locales-registry.mjs",
```
So the registry regenerates on every `npm run dev` / `npm run build`.
Hand-edit a JSON dictionary → restart dev or run `npm run locales:build`
to pick up changes.

**Composable returns a ref, template needs deref:** in templates,
`useTranslations('ErrorPages').value.backToQuotes` doesn't auto-unwrap
through nested property access. Use a computed deref:
```ts
const errorPagesLabels = useTranslations('ErrorPages');
const errorBackLabel = computed(() => errorPagesLabels.value.backToQuotes);
```
then bind `:backLabel="errorBackLabel"`.

**Provider env var:** `app/lib/i18n/index.ts` reads
`process.env.NUXT_PUBLIC_TRANSLATIONS_PROVIDER` (defaults to 'file').
The seam is there for a future remote-provider variant; today only
'file' is implemented.

**Path differs from propeller-vue:** propeller-vue's mirror lives
under `src/locales/`. Nuxt's `srcDir: 'app/'` means the path is
`app/locales/` here. The build script writes the path explicitly —
DO NOT swap `LOCALES_DIR` back to `src/locales` thinking it's a typo.

**History:** before 2026-06-08 `useTranslations` was a stub that
returned `{}` for every namespace. Every page was already calling
`useTranslations('Foo')` correctly; the package components were just
falling back to their own English defaults because the namespace
maps were empty. The 2026-06-08 i18n commit swapped the stub for
the file provider and back-filled the locales dir — no page-side
wiring changes were needed. Lesson: when porting i18n elsewhere,
check the consumer's existing `useTranslations` calls first; the
binding side may already be in place.
